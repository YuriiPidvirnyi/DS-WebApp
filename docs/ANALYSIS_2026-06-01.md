re-verified in code and is marked accordingly.

## 3. Architecture

Modern, cleanly assembled Next.js 16 App Router monolith.

- Thin `app/**` pages wrap real content in `src/views/**`.
- 48 API routes in 17 groups; 4 Vercel crons (notifications/5m, reminders, recall, low-stock).
- Integrations with graceful fallback: Supabase (auth+DB), Upstash Redis (cache + rate-limit, in-memory
  fallback), Resend (email), Monobank (payments, ECDSA webhook), Cloudflare Turnstile, Sentry, Vercel AI Gateway.
- Platform security in `proxy.ts` middleware: per-request nonce CSP, HSTS (2y), X-Frame-Options DENY,
  COOP, Permissions-Policy + Supabase session refresh and route guards for `/admin`, `/cabinet`.
- i18n: uk eager, en/pl lazy. Clever `.ts` wrappers over `.json` to defeat Turbopack tree-shaking,
  guarded by parity/drift/completeness tests.
- PWA: next-pwa + Workbox with per-type cache strategies and an offline fallback.

**Observation:** every route in the build is dynamic (`ƒ`), including marketing `/` and `/about`.
Partial static generation for public pages would help SEO/LCP and CDN caching.

## 4. Database — authoritative live data

Project `exgpwtyrkkhwqqdgqbkz`, **Postgres 17**, region **us-east-1**.

> **Cross-region latency:** the app runs in Vercel `fra1` (Frankfurt) while the DB is in `us-east-1`
> (USA). Every app↔DB round trip crosses the Atlantic (~90–100 ms RTT). For a Ukrainian clinic both
> should be in the EU.

**Live Security Advisor: 72 warnings, 0 ERROR** (no RLS-disabled tables, no critical holes):

| Lint                                                 | Count | Note                                                                                                                                                              |
| ---------------------------------------------------- | ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `*_security_definer_function_executable` (anon/auth) | 60    | mostly informational (helper functions)                                                                                                                           |
| `function_search_path_mutable`                       | 6     | `deduct_inventory`, `add_inventory`, `admin_doctor_id`, `admin_can_read_appointment`, `set_payment_configs_updated_at`, `next_req_number` — add `SET search_path` |
| `rls_policy_always_true`                             | 4     | `payments` (insert_anon, update_service_role) + `patient_wallet_cards` (insert/update service)                                                                    |
| `public_bucket_allows_listing`                       | 1     | bucket `material-images` allows listing all files                                                                                                                 |
| `auth_leaked_password_protection`                    | 1     | HaveIBeenPwned check disabled — enable in dashboard                                                                                                               |

**Live Performance Advisor: 168 (100 WARN / 68 INFO):**

- 36 × `auth_rls_initplan` — RLS policies call `auth.uid()`/`auth.jwt()` un-wrapped, causing per-row
  re-evaluation. Wrap as `(select auth.uid())`. **Highest-value perf fix.**
- 38 × `unindexed_foreign_keys` — FKs without a covering index.
- 62 × `multiple_permissive_policies` — planner overhead from multiple permissive policies.
- 2 × `duplicate_index` on `stock_document_items` — exact duplicates, drop one of each.
- 30 × `unused_index` — this is low-traffic **preprod**; do NOT drop based on this.

> The advisors confirm the live DB contains tables absent from the working branch's migrations
> (`patient_wallet_cards`, `internal_requisitions(+items)`, `inventory_audit_items`, `clinic_settings`,
> `stock_document_items`) — proving the prod DB = develop state (v3.3.x), ahead of main.

**Schema notes (static):** `treatment_records.created_by` written by seed but no column/FK in branch;
`appointment_reminder_preferences.appointment_id` is TEXT not FK; messy doctor-scope RLS fix cycle
(v1 → v2) leaves orphan migrations. Strong points: RLS on all tables, full audit trail with restore,
atomic `deduct/add_inventory`, partial-unique dedup indexes.

## 5. Security — verified, with corrections

Overall security-conscious: CSRF (custom header), nonce CSP, server-side Turnstile, Redis rate-limit,
ECDSA Monobank webhook verification, AI token caps, GDPR export/soft-delete, server-only secrets, gitleaks.

Two "critical/medium" automated findings were **re-verified and downgraded**:

- ❌ "RLS uses JWT `is_admin` claim → privilege escalation" — **false for the current state.**
  Migration `20260322` redefined `is_admin_actor()` to membership-only
  (`auth.uid() IS NOT NULL AND EXISTS(admin_users)`); `is_admin()` is membership-only too. The JWT-claim
  policies in the baseline migration are superseded by later ones, and the **live advisor reports no such hole.**
- ❌ "no cookie consent (GDPR)" — **false.** `CookieConsent.tsx` + `ConsentGateAnalytics.tsx` exist and
  GA4/Vercel/Sentry Replay are gated behind consent.
- ✅ Real but WARN-level: `payments_insert_anon WITH CHECK(true)` (recommend `appointment_id IS NOT NULL
AND amount_kopecks > 0`); rate-limit is per-IP not per-user; CRON_SECRET has no rotation; `material-images`
  bucket lists; leaked-password protection disabled.
- ✅ `/api/e2e/auth-links` is correctly gated (`NODE_ENV !== 'production'`).

**Net security priority: medium.** No active exploit vector; the backlog is hardening.

## 6. Backend / API

Consistent patterns: `requireRole()`/`requirePermission()` on admin routes, CSRF on mutations, Zod on
public forms, Turnstile on public POSTs. Crons are idempotent (atomic `processing` claim + partial-unique
index against duplicate reminders). Recall (3-touch) via `get_recall_candidates` RPC.

Minor real issues: `/api/payments/create` lacks an idempotency key (a repeated POST creates two invoices);
`/api/admin/ai-usage` has no explicit rate limit; the low-stock cron silently reports success when
`ADMIN_NOTIFICATION_EMAIL` is unset.

## 9. Dependencies & vulnerabilities

`npm audit` on the working branch: **12 vulnerabilities (8 moderate, 4 high)**:

- Next.js 16.1.6 — high: DoS in Server Components + **App Router middleware/proxy bypass** (exactly where
  auth + security headers live) + redirect cache-poisoning.
- `axios` (auth-bypass prototype pollution, NO_PROXY bypass), `@babel/plugin-transform-modules-systemjs`,
  `fast-uri` (path traversal).
- moderate: `ws`, `postcss` (XSS), `uuid`→`svix`→`resend`, `follow-redirects`, `brace-expansion`.

This is on main (29 behind). develop's dependabot is already raising versions (PR #295 + a series of
minor/patch bumps). **Prioritize merging the dependabot bumps and raising Next.js**, since the
middleware bypass directly touches this app's security model.

## 11. Verdict

A mature, thoughtfully built, security-conscious product at "production-ready" level. Strong
architecture, clean code (build/lint/types/tests all green, 0 ERROR in the live security advisor). The
biggest risk is process/infrastructure, not code: main↔develop↔live-DB desync, two huge divergent
branches, stale docs, and the transatlantic DB region. The initially "critical" vulnerabilities were
re-verified and found over-stated; the real security backlog is medium-priority hardening plus an urgent
Next.js upgrade.
